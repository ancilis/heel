# HEEL — Red-team findings (v1 safety-spine adversarial review)

4-agent workflow attacking the auth gate, prompt-injection surface, §10.2 conduct guarantees,
and backtest honesty. The #1 claim HELD (a prompt-injected MCP caller cannot escape a signed
scope). Gaps between claim and implementation were fixed — see EVAL §8 / DECISIONS D-009..D-014.

## Red-team adversary attacking the scope-escalation boundary (the #1 safety claim, §10.1)

**Verdict:** CONTAINED on the stated #1 claim (a prompt-injected MCP/REST caller cannot create, widen, add a target to, or run outside the signed allowlist — every escalation path I tried fails closed), BUT the dependent conduct/containment sub-claim is BROKEN: rate/resource limits are not enforced. Plus a medium key-management weakness undermines the tamper-evidence story against a local adversary.

### [HIGH] Rate/resource limits are stored and signed but never enforced at runtime
*Location:* `/Users/hellohelloalbus/heel/heel/orchestrator.py:30`

**Evidence:** Created a scope with rate_and_resource_limits={'max_requests':1} and ran heel_run against the real target 'synthetic-saas' 20 times — all 20 returned status='complete'. grep across heel/ shows rate_and_resource_limits is only WRITTEN (scope.py:71,76), put in the canonical signature (scope.py:51), exposed in public_view (contracts.py:157), and LOGGED once (orchestrator.py:30 'limits': scope.rate_and_resource_limits). It is never READ by run_abuse (orchestrator.py:24-53) or run_adversarial (agents.py:138-181). contracts.py:146 comments the field as 'enforced server-side regardless of caller request' — that enforcement does not exist.

**Recommendation:** Implement server-side enforcement in run_abuse/HeelServer: a per-scope request counter (persisted in Store keyed by scope_id) checked against max_requests before each run, plus max_concurrency and backoff honored in the probe loop. Until enforced, either remove the 'enforced server-side' claim from contracts.py:146 and the §10 conduct claim, or gate runs to fail closed when a limit field is present but unhandled. Add a test that asserts the Nth run beyond max_requests is rejected.

### [MEDIUM] HMAC signing key co-located with the scopes it protects; local-write adversary can forge/widen a valid scope
*Location:* `/Users/hellohelloalbus/heel/heel/scope.py:32`

**Evidence:** _paths() places signing.key and scopes/ in the same HEEL_HOME directory (scope.py:27-29). Empirically: after create_scope, I loaded the scope file, appended 'target-EVIL' to target_allowlist, recomputed signature = _sign(_canonical(d)) using the on-disk key, and verify() returned (True,'ok'). So tamper-evidence depends only on filesystem write permission to HEEL_HOME, not on possession of an out-of-band secret — any actor that can write a scope file can also read the key and mint/widen valid scopes. Additionally scope.py:36 claims the key is 'stable per repo' but it is derived from os.urandom (non-deterministic), a docstring/impl mismatch.

**Recommendation:** Move the signing key out of HEEL_HOME to an operator-controlled location (env var pointing outside the data dir, OS keychain, or HSM/KMS), so the trust boundary is 'possess the key' not 'can write the scopes dir'. Document the threat model honestly: HMAC tamper-evidence only protects against actors WITHOUT the key. Fix the os.urandom-vs-'stable' docstring. This does not affect the MCP/REST agent-caller claim (the agent controls no filesystem path), so keep the #1-claim messaging but scope it to remote callers.

### [INFO] Allowlist check passes for target strings that are not real targets (defense-in-depth observation, not exploitable)
*Location:* `/Users/hellohelloalbus/heel/heel/scope.py:143`

**Evidence:** target_in_scope (scope.py:143-144) is a literal membership test against the signed allowlist. If a human signs a scope whose allowlist contains a non-existent id (e.g. 'saas' instead of 'synthetic-saas'), heel_run passes the auth gate (mcp_server.py:99) and is only stopped later by get_target() returning None (orchestrator.py:33-36). No escalation results, but it means the allowlist is not validated against the known target registry at sign time.

**Recommendation:** Optionally validate target ids against TARGETS (targets.py:134) during create_scope so an operator cannot sign a typo'd/garbage allowlist entry. Purely hardening; not a boundary break.

## §10.2 conduct-guarantee auditor (weaponization, content, PII, ranking, severity, audit-log integrity, lane discipline)

**Verdict:** CONDITIONAL PASS for v1-as-synthetic-demo (no real weaponization/content/PII path exists because all targets are stubs and probes only read declared properties). FAIL on two §10.2 guarantees as written: (1) the immutable/tamper-evident audit log is NOT tamper-evident against an adversary with store access — full re-chain, tail-truncation, and whole-run deletion all pass verify_chain; runs can be unattributed; (2) "limits enforced server-side regardless of caller" is false — only allowlist/expiry/signature are enforced; rate/resource/budget limits are logged but never applied. Plus severity-inflation and fragile plausibility-demotion. None are exploitable into real-world harm in v1 (synthetic stubs), but every one of them becomes load-bearing the moment Phase-3 wires a real target or a live LLM probe loop, and "contained" is asserted, not enforced.

### [HIGH] ContainmentLog is chained but unsigned/unanchored: attribution can be rewritten and the whole log re-chained while verify_chain still reports VALID

**Evidence:** containment.py:28-39 computes entry_hash=sha256(prev_hash+canonical(entry)) using only public, recomputable inputs; there is NO HMAC/signature over entries and no external anchor (no signed head, no checkpoint), even though scope.py:58-59 already has an HMAC signing key available. verify_chain (containment.py:46-60) only re-derives the same SHA-256 chain, so anyone with store write access can recompute it. Empirically confirmed: I changed every caller='attacker-agent' to 'innocent-bystander', re-chained all 78 rows, and verify_chain returned (True, '78 entries verified'). The §10.2.8 / module-docstring claim 'any tampering with the audit trail is detectable' is false against an adversary who can write the containment table (the same trust boundary SQLite gives the host process). The shipped test (tests/test_heel.py:164-171) only tests a naive single-row edit WITHOUT re-chaining, so it passes while the real attack succeeds.

**Recommendation:** HMAC-sign each entry (or at least a periodic signed checkpoint of the head hash) with the existing signing.key, the same way scopes are signed; verify the signature in verify_chain. Store the latest head hash in an append-only/external location so a full re-chain cannot pass. Without a secret in the chain, SHA-256 chaining provides integrity only against parties who CANNOT recompute it — which is nobody with DB access.

### [HIGH] Tail-truncation and whole-run deletion are undetectable; a 'complete' run with zero log entries passes verification (run can avoid being logged/attributed)

**Evidence:** verify_chain (containment.py:46-60) walks rows in seq order and only checks adjacent linkage; it never asserts the chain reaches the current head or covers an expected sequence, so deleting the last entry leaves it valid (confirmed: after DELETE of MAX(seq) it returned True/'77 entries verified'). Worse, store.add_run (store.py:32-36) can create/replace a run row independently of the log; I inserted run-ghost with status='complete' and no containment rows, and verify_chain(store,'run-ghost') returned (True, '0 entries verified') with an empty log. heel_get_containment_log (mcp_server.py:133-136) surfaces this as chain_valid=True. So a completed run can exist with NO attribution and the audit self-report still says 'valid'. orchestrator.run_abuse always logs run_start (orchestrator.py:29) on the happy path, but nothing cross-checks that every 'complete' run HAS a run_start/run_complete pair, and nothing detects deletion of those rows.

**Recommendation:** Verify completeness, not just linkage: assert seq is contiguous from 0 (or from a signed prior checkpoint) to a signed head; reject runs whose run_id has no run_start+run_complete pair in the log; treat '0 entries verified' for a known run_id as a FAILURE, not success. Persist a monotonically increasing signed high-water seq so truncation is detectable.

### [MEDIUM] 'contained / canary_only' is a hard-coded label in the finding, not an enforced containment mechanism

**Evidence:** agents.py:124-132 _vector sets reproduction={... 'sample':'canary_only','contained':True} unconditionally for every finding; the probe evaluators (_p_* agents.py:36-64) merely read declared affordance properties and return an evidence dict. There is no egress filter, sample cap, byte limit, time/iteration ceiling, or containment monitor anywhere; ContainmentScenario.containment_limits (contracts.py:80) is defined but never consulted. The §10 docstring (agents.py:10-13) asserts 'no real exfiltration, no sustained exhaustion' — true in v1 ONLY because targets are in-memory stubs (targets.py) and probes never touch a network. The moment a real probe body is added in Phase 3, 'contained=True' will still be stamped on the finding regardless of what the probe actually did. The test (test_heel.py:145-148) only asserts the literal label, reinforcing label-as-proof.

**Recommendation:** Make containment a runtime invariant, not a string: route every probe through a sandbox that enforces synthetic-only data handling (DataHandlingMode), an egress allowlist, and a per-scenario sample/byte/iteration cap from containment_limits; set contained=True only if the enforcement layer attests it. Otherwise the #1 safety claim is unfalsifiable.

### [MEDIUM] Rate/resource limits and the run budget are enforced for NOTHING — contradicting 'enforced server-side regardless of caller request'

**Evidence:** contracts.py:146 documents rate_and_resource_limits as 'enforced server-side regardless of caller request', and scope creation defaults them (scope.py:71). But grep of agents.py+orchestrator.py shows max_requests/max_concurrency/backoff/sleep each appear 0 times; rate_and_resource_limits appears once — only as a value LOGGED at run_start (orchestrator.py:29-30). The budget arg accepted by heel_run (mcp_server.py:45-46,107) is never passed to or read by run_adversarial. Only target_allowlist, signature, and expiry are actually enforced (mcp_server.py:88-104). So the denial-of-wallet / sustained-exhaustion guarantee rests on an un-enforced field. No real exhaustion is possible in v1 (the probe loop is bounded by the finite synthetic affordance list), but the guarantee as written is false and will silently fail against a real target.

**Recommendation:** Actually enforce limits in the orchestrator/agent loop: a server-side request counter that aborts at max_requests, concurrency gating at max_concurrency, and backoff; reject or clamp the caller-supplied budget against the signed scope. Until then, drop the 'enforced server-side' wording so reviewers don't over-trust it.

### [MEDIUM] Severity inflation: content-guardrail-absent forces a hard likelihood 0.9 / impact 1.0 (critical) overriding the scenario's real priors

**Evidence:** agents.py:60-64 _p_content_guardrail returns 'max_severity':True whenever blocks_prohibited is not True; agents.py:119-120 then overrides to likelihood,impact=0.9,1.0 (score 0.9 => 'critical' per contracts.py:99-101) regardless of scenario.severity_model. Confirmed: with a scenario whose priors were {0.1,0.1}, the finding came out 0.9/1.0/critical. The Severity.uncertainty band (contracts.py:92, 'no inflation, §10.2.7') is never populated or surfaced in scoring. The trigger is attacker/target-controlled data (the blocks_prohibited property), so a hostile or sloppy target definition can mint a guaranteed top-ranked 'critical' from a single boolean, crowding the queue and distorting severity_calibration.

**Recommendation:** Replace the hard 0.9/1.0 with the scenario's modeled impact and an explicit, surfaced uncertainty band; if a missing content guardrail truly warrants high impact, encode that in that scenario's severity_model rather than a global override, and cap how much any single boolean can move rank.

### [MEDIUM] Plausibility-demotion is load-bearing but data-cooperative: a truly-unreachable affordance is ranked CRITICAL unless it self-declares unreachability via two specific property keys

**Evidence:** estimate_reachability (agents.py:103-111) returns 0.15 (below PLAUSIBILITY_FLOOR 0.25) ONLY when properties contain client_reachable is False OR a 'requires' key; otherwise it returns 0.78 (or 0.55 if guarded). The true planted reachability field (Affordance.reachability, contracts.py:183) is NEVER read by the agent (only by the backtest, targets.py:48). Confirmed: an affordance with true reachability 0.05 but a 'tenant_check':missing property and no client_reachable/requires keys was scored plausible=True, reachability 0.78, severity 0.72, label critical. The repo's degenerate decoy legacy_import (targets.py:89-90) is demoted only because it happens to set BOTH client_reachable=False and requires. So 'degenerate findings are flagged, never ranked' (contracts.py:16) holds for the cooperative demo data but is not robust — a deep-chain-only finding that doesn't advertise those two keys lands as a top risk. Also note discovery_pass (agents.py:188-211) filters aff.decoy but not reachability, so it can surface implausible affordances too.

**Recommendation:** Derive the agent's reachability estimate from a richer observable model (depth/gating/auth-step count) and cap it for affordances showing high prerequisite depth, instead of a single 0.78 default gated on two string keys; have discovery_pass apply the same plausibility floor before emitting a finding.

### [LOW] Lane discipline holds on tested paths, but the jailbreak/content surfaces escape the discovery weaponizer only by data coincidence (structural fragility)

**Evidence:** Positive: the SSRF affordance keeps handoff_to_appsec=True even when surfaced via discovery_pass (agents.py:123, _vector reads aff.properties['handoff']=='appsec'; confirmed url_fetch came out handoff_appsec=True through the discovery path), and the pure-jailbreak surface is short-circuited to a handoff before any vector is built (agents.py:162-166). However, discovery_pass (agents.py:188-211) decides whether to weaponize an affordance purely by scanning property KEYS for _CONTROL_HINT_KEYS substrings with a missing value; agent_prompt_surface (jailbreak) and content_guardrail escape it only because their property dicts happen not to contain such a key (confirmed both excluded). There is no explicit rule like 'never emit a product-abuse finding for a jailbreak_surface or content_guardrail affordance' — if a future target adds, say, isolation:'missing' to a jailbreak surface, discovery_pass would mint a product-abuse finding with no model_redteam handoff. Lane discipline is therefore enforced by the shape of the demo data, not by an invariant.

**Recommendation:** Add an explicit lane guard in discovery_pass (and _vector): if an affordance carries jailbreak_surface or kind=='content_guardrail' or handoff in {appsec,model_redteam}, route to the corresponding handoff and never emit it as a weaponized product-abuse finding, regardless of which property triggered discovery.

### [INFO] No real-PII-harvest path in v1, but the data-handling mode that would gate it is never enforced

**Evidence:** There is no PII source anywhere: targets.py affordances carry only synthetic metadata (routes, flags, booleans), probes read those properties (agents.py:36-64), and findings persist only canary evidence (store.py:46-52). DataHandlingMode.SYNTHETIC_ONLY is the signed default (contracts.py:62-64, scope.py:66), so the intended guarantee exists. But like the resource limits, data_handling_mode is signed and surfaced yet NEVER consulted by the agent/probe/containment code — confirmed by absence from agents.py/orchestrator.py logic. So today there is genuinely no PII harvest, but the control that is supposed to prevent it under MINIMIZE/real-target mode is not wired to anything.

**Recommendation:** When real targets land, gate every probe's data handling on scope.data_handling_mode (refuse to persist or return record contents unless mode permits, redact by default). Add a test that a MINIMIZE/real run cannot persist record contents.

## measurement skeptic / backtest auditor

**Verdict:** CIRCULAR (with honest scaffolding) — coverage/calibration metrics are self-consistency checks, not detection-accuracy evidence. Trustworthy only as a wiring/contract demo; misleading if cited as 0.9+ accuracy. Needs blind-target + held-out-scenario evaluation before any real-target claim.

### [HIGH] Seed scenarios are ~1:1 with planted vectors; probes key off the literal property strings the plant sets (the coverage is hand-matched, not discovered)

**Evidence:** 10 of 12 reachable planted SaaS vectors have exactly one dedicated firing seed scenario (export_records->sc.export.entitlement, usage_meter->sc.meter.reset, record_get->sc.record.tenant, premium_toggle->sc.flag.tiergate, trial_signup->sc.trial.serial, admin_purge->sc.audit.coverage, internal_debug->sc.endpoint.hidden, referral_credit->sc.referral.selfdeal, account_recovery->sc.identity.recovery; webhook via discovery; promo_stacking missed). The probe condition IS the plant's property literal: heel/agents.py:40 `_p_tenant` returns a hit iff `properties.get('tenant_check')=='missing'`, and heel/targets.py:64 sets `tenant_check='missing'` on record_get. Same for gated_by=='client' (agents.py:41 / targets.py:66), reset_window=='client_controlled' (agents.py:39 / targets.py:62), audit_logged is False (agents.py:43 / targets.py:70), recovery_check=='weak' (agents.py:49 / targets.py:76). The success_criterion in scenarios.py (e.g. line 46 {'prop':'tenant_check','equals':'missing'}) is the same string again. So the 0.91/0.94 measures 'the dict value I wrote still equals the dict value I wrote.' Of the headline coverage, essentially ALL of it is hand-matched; only 1 vector (webhook) is found by a non-seed path, and 1 (promo_stacking) is deliberately missed.

**Recommendation:** Stop citing 0.9+ as accuracy. Re-frame this backtest as a wiring/self-consistency test. For a real metric, the planted weakness and the probe's observable must be authored INDEPENDENTLY (different people, or generated targets the scenario authors never see), and probes must infer abuse from indirect/observable signals (HTTP behavior, response diffs) rather than reading a property whose name telegraphs the answer (`tenant_check`, `self_referral_check`). Report coverage on held-out scenario families (leave-one-category-out).

### [HIGH] Severity calibration 0.78 is inflated: 7 of 11 scenario priors are set EXACTLY equal to the planted ground-truth severity

**Evidence:** Verified pairs (scenario prior vs planted true_severity, likelihood,impact): sc.export.entitlement (0.7,0.8)==(0.7,0.8); sc.record.tenant (0.8,0.9)==(0.8,0.9); sc.flag.tiergate (0.6,0.7)==(0.6,0.7); sc.trial.serial (0.5,0.5)==(0.5,0.5); sc.audit.coverage (0.6,0.7)==(0.6,0.7); sc.identity.recovery (0.6,0.7)==(0.6,0.7); sc.endpoint.forced (0.3,0.4)==(0.3,0.4) — 7 EXACT. The agent copies the scenario prior straight into the finding (heel/agents.py:117-122 sets likelihood/impact from `scenario.severity_model`), and the backtest then correlates that against the plant (heel/backtest.py:78-81). For the SaaS run, pred_sev==true_sev exactly on 7 of 10 TPs; over the only 3 pairs where they DIFFER the Spearman is 1.0. So the 0.775 is dominated by self-correlation of priors the author tuned near ground truth, not by HEEL inferring severity.

**Recommendation:** Set scenario priors WITHOUT knowledge of the planted true_severity (or have a different author set them), and add deliberate prior/ground-truth disagreement so the metric can actually drop. Report severity calibration only over the subset where prior != ground-truth. With n=10 (SaaS) / n=16 (AI), also report a confidence interval — Spearman at n=10 has a ~0.55 std error, so 0.78 is not distinguishable from 0.4.

### [MEDIUM] Reachability-weighting is cosmetic for the headline and does NOT demote the degenerate; the degenerate is excluded twice before weighting applies

**Evidence:** Measured cov_w - cov = 0.009 (SaaS) and 0.006 (AI) — the weighting changes nothing visible. The weight numerator (heel/backtest.py:69) uses the matched finding's reachability_score, but `estimate_reachability` (heel/agents.py:103-111) returns only ~4 constant values (0.15, 0.55, 0.78, 0.83); every TP gets ~0.78, so weighting is uniform. The degenerate (legacy_import) is NOT demoted by weighting at all: it has plantedReach=0.12 < PLAUSIBILITY_FLOOR=0.25, so it is dropped from `reachable[]` at backtest.py:56 (never in the denominator, not even counted as a miss), AND the agent's estimate 0.15<floor filters it from `plausible` findings at backtest.py:55. The 'plausibility-weighting bait' is actually handled by two hard threshold cutoffs, not by the weighted-coverage formula.

**Recommendation:** Either make reachability-weighting load-bearing (give it a continuous estimator that varies across findings and visibly moves cov_w away from cov) or drop the cov_w column and state plainly that the degenerate is handled by a plausibility floor. Don't present cov_w as if reachability scoring is doing work it isn't.

### [MEDIUM] Attacker/real-target can trivially evade the reachability demotion — it recognizes exactly two magic keys

**Evidence:** estimate_reachability (heel/agents.py:104-111) demotes to 0.15 ONLY if `client_reachable is False` OR a `requires` key is present; otherwise it returns 0.55 (guarded) or 0.78 (unguarded). The degenerate sets BOTH magic keys (targets.py:90: client_reachable=False, requires='5-step chain'). Verified: an affordance with `depth=50` (truly unreachable but using a different key) still scores 0.78 = plausible. So the degenerate is demoted because it was authored to trip the exact keys the estimator checks. A real low-reachability surface encoded any other way is scored fully plausible, and an attacker wanting to suppress a finding cannot rely on this — but more importantly the demotion logic has no generality.

**Recommendation:** Reachability must be estimated from probe-observed evidence (call depth actually traversed, auth steps actually required during the contained probe), not from trusting two declared property keys. Test it against degenerates that hide unreachability in varied ways.

### [MEDIUM] FP rate 0.09/0.06 rests on a single firing decoy; the other decoy never fires, so the FP denominator-of-bait is effectively n=1

**Evidence:** Two decoys are planted (targets.py:84 export_billing, targets.py:86 admin_rotate_keys). Only export_billing fires — via the one deliberately over-broad heuristic sc.export.overbroad (scenarios.py:38, route_contains 'export'). admin_rotate_keys fires on NO scenario (its kind admin_action only meets sc.audit.coverage, which requires audit_logged is False, and the decoy sets audit_logged=True). So the FP rate is driven entirely by one decoy caught by one intentionally-loose probe. FP rate = 1/11 = 0.09 is a single data point, not a rate. The system isn't shown to resist FPs broadly; it's shown to produce exactly the one FP the author engineered.

**Recommendation:** Add many hardened-but-tempting decoys across every category (10+), including decoys that share property names with vulnerable affordances but have safe values, and decoys specifically targeting each probe's heuristic. Only then is an FP rate meaningful. Report it with a CI.

### [LOW] Category-10-clean-on-non-AI is structural/tautological, not a discovered safety property

**Evidence:** cat10=0 on SaaS is guaranteed by TWO independent constructions: (a) all 7 category-10 scenarios are gated applies_when=HAS_AGENT_SURFACE (scenarios.py:71-99) and skipped at heel/agents.py:146 when target.has_agent_surface is False; AND (b) the SaaS target plants zero agent_tool/mcp_connector affordances (verified kinds: admin_action, auth_reset, endpoint, export, flag, integration, meter, record, referral, trial). With both gates, cat10=0 cannot be anything but 0 — it's a tautology, not evidence that 'category 10 is cleanly optional' under adversarial conditions. It would only be informative if a category-10 probe COULD fire on a non-agent surface and was shown not to.

**Recommendation:** To make this claim non-trivial, run category-10 scenarios UN-gated against the SaaS target and show they still produce 0 findings because the affordance kinds genuinely don't exist / don't match — that tests the probes' specificity rather than the applies_when flag. As written, it only confirms the flag is read.

### [INFO] The genuine FN (promo_stacking) and the swarm-discovery (webhook) are both artifacts of the missing-control encoding convention, not emergent results

**Evidence:** Both the seed probes and the discovery pass (heel/agents.py:196-197) detect abuse ONLY via a property whose key contains a control-hint substring ('protection','check','filter','isolation','guard','logged','allowlist','limit') AND whose value is in {missing,none,false,False}. webhook_endpoint is 'discovered' purely because it sets replay_PROTECTION='missing' (matches 'protection' + missing). promo_stacking is the honest FN purely because its weakness is encoded as a PRESENT-positive (stackable=True, client_reachable=True) with no missing-control key — verified none of its props match. So the FN/discovery split is determined by how the author chose to encode each weakness (missing-control vs present-positive), not by any difference in how 'hard' the vectors are to find. The honesty markers are well-intentioned and improve the demo, but they don't generalize.

**Recommendation:** Keep these as demonstration honesty markers but label them as such. Real evidence of swarm discovery requires the discovery pass to find a weakness whose encoding it was NOT designed to pattern-match (e.g., infer coupon-stacking from observed probe behavior). Add a held-out 'present-positive' weakness family the discovery pass is expected to catch to prove it isn't limited to the missing-control convention.

### [INFO] What would make the backtest trustworthy for real targets (concrete protocol)

**Evidence:** Current loop: same author writes targets.py (plants) and agents.py/scenarios.py (probes) keyed on identical dict literals; backtest.py scores agreement. No blind set, no held-out scenarios, no independent severity ground truth, n=10-16.

**Recommendation:** (1) BLIND TARGETS: generate/commission synthetic targets by a party who never sees the scenario library; scenario authors never see the plants. Probes must read only behavior observable through the contained PoC, not telegraphing property names. (2) HELD-OUT EVAL: leave-one-category-out and leave-one-scenario-out coverage; report degradation. (3) INDEPENDENT SEVERITY: have a separate rater assign ground-truth severity; force prior!=truth on a known fraction; report calibration only where they disagree, with CIs. (4) ADVERSARIAL DECOY SUITE: 10+ hardened decoys per category, including name-collision decoys, to make the FP rate real. (5) NEGATIVE CONTROLS: a fully-hardened target (expect coverage undefined/0 findings, FP=0) and a target with weaknesses in categories that have NO seed scenario (measures true generalization gap). (6) SAMPLE SIZE: n>=50 planted vectors per target before quoting any rate; report bootstrap CIs on coverage/FP/calibration. (7) REAL DOGFOOD: run against one real internal product with weaknesses found independently (e.g., by a human red team) and report recall against that human-found set — that is the only number that rebuts circularity.
